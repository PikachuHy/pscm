# Guile 1.8 Feature Audit for pscm

**Date**: 2026-05-02
**Status**: in-progress

## Special Forms

| Special Form | Status | Implementation | Notes |
|-------------|--------|---------------|-------|
| `define` | present | `src/c/define.cc:eval_define` | Handles `(define var expr)`, `(define (name args...) body...)`, and curried define forms. Variable bindings respect lexical scoping (nested/lambda envs vs top-level/module). |
| `lambda` | present | `src/c/eval.cc:eval_lambda` (line 123) | Supports normal parameter lists, dotted rest params `(lambda (x . y) ...)`, and single-symbol rest params `(lambda x ...)`. |
| `if` | present | `src/c/eval.cc:eval_if` (line 167) | Standard conditional with consequent and optional alternative. Returns unspecified value (none) when no else branch. |
| `cond` | present | `src/c/cond.cc:eval_cond` | Supports `else` clause, `=>` arrow syntax (uses `is_literal` heuristic similar to Guile's `literal_p`), and normal expression sequences. |
| `case` | present | `src/c/case.cc:eval_case` | Supports `else` clause and multiple datum comparisons via `eqv?`. Returns `#f` when no clause matches. |
| `do` | present | `src/c/do.cc:eval_do` | Full implementation with parallel binding semantics (all step expressions evaluated before any variables are updated). Supports optional step expressions and body expressions. |
| `let` | present | `src/c/let.cc:expand_let` | Implemented as source-to-source transformation to `((lambda ...) ...)`. Supports named let with full correct semantics (args evaluated in outer env before letrec binding). |
| `let*` | present | `src/c/let.cc:expand_letstar` | Implemented as source-to-source transformation to nested `let`. |
| `letrec` | present | `src/c/let.cc:expand_letrec` | Implemented as source-to-source transformation to `(let ((var #f)) (set! var ...) ...)`. |
| `and` | present | `src/c/eval.cc:eval_and` (line 183) | Short-circuit evaluation; returns `#t` with no arguments, the last expression's value otherwise, or `#f` on first falsy. |
| `or` | present | `src/c/eval.cc:eval_or` (line 208) | Short-circuit evaluation; returns `#f` with no arguments, the first truthy value, or the last expression's value. |
| `delay` | present | `src/c/delay.cc:eval_delay` | Creates a promise (zero-arg thunk). `force` (`scm_c_force`) implements R5RS-compliant re-entrant semantics: if a promise is forced while already being forced, the inner force caches the value. |
| `quasiquote` | present | `src/c/quasiquote.cc:eval_quasiquote` | Implements `unquote`, `unquote-splicing`, nested quasiquote, and vector quasiquote (`#( ... )`). Uses direct evaluation approach (similar to Guile 1.8's `iqq`). |
| `quote` | present | `src/c/eval.cc:eval_quote` (line 65) | Simple: returns the first argument unevaluated. |
| `set!` | present | `src/c/eval.cc:eval_set` (line 71) | Searches local environment first (no parent search), then parent environments, then current module. Creates new binding if not found anywhere. |
| `begin` | present | `src/c/eval.cc` inline (line 424) | Evaluates all sub-expressions in sequence, returns the last value. Returns `#f` when no sub-expressions (unspecified in R5RS). |
| `define-macro` | present | `src/c/macro.cc:eval_define_macro` | Explicit renaming macros. Supports both `(define-macro (name args...) body...)` and `(define-macro name transformer)` forms. Handles rest parameters in macro args. |
| `define-module` | partial | `src/c/module.cc:eval_define_module` | Creates modules and registers them. Supports options: `#:use-module`, `#:export`, `#:pure`, `#:autoload`, `#:no-backtrace`. Option parsing noted as "simplified version" (comment on line 726). |
| `use-modules` | present | `src/c/module.cc:eval_use_modules` | Resolves module names and adds their public interfaces to the current module's uses list. |
| `export` | present | `src/c/module.cc:eval_export` | Marks symbols for export from the current module. |
| `re-export` | partial | `src/c/module.cc:eval_re_export` | Re-exports symbols from other modules. Very lenient -- allows re-exporting symbols not found in the module (comment on line 908: "allows re-exporting symbols that will be available later"). A stricter implementation would error on missing symbols. |
| `define-public` | present | `src/c/module.cc:eval_define_public` | Combines `define` with `export`. |
| `call/cc` / `call-with-current-continuation` | present | `src/c/eval.cc:eval_call_cc` (line 233) | Implemented via `setjmp`/`longjmp` in `continuation.cc`. Handled as both a special form (symbol dispatch) and a function value (C++ function dispatch at line 593). Stack data is copied on capture, restored on invocation. |
| `apply` | present | `src/c/apply.cc:eval_apply` | Handles procedure, function, and continuation application. Supports the standard `(apply proc arg1 ... args-list)` syntax. Contains special-case handling for nested `apply`/`map`/`call/cc` application (lines 331-392). |
| `map` | present | `src/c/map.cc:eval_map` | Works with procedures and functions. Supports multiple list arguments (shortest list determines length). Note: heap-allocates `list_ptrs` array (line 57), which may interact poorly with continuations. |
| `for-each` | partial | `src/c/for_each.cc:eval_for_each` | Hard-coded `MAX_FOR_EACH_ARGS = 10` limit (line 59). Otherwise complete: supports procedures and functions, multiple lists, proper length validation. |
| `map-in-order` | present | `src/c/map.cc:eval_map` | Alias for `map` (line 456 in eval.cc dispatches `map-in-order` to `eval_map`). In pscm, `map` already preserves order. |
| `call-with-values` | present | `src/c/values.cc:eval_call_with_values` | Calls producer, passes values list to consumer. `values` implemented as a C function (`scm_c_values`) that returns its arguments as a list. |
| `dynamic-wind` | present | `src/c/dynwind.cc:eval_dynamic_wind` | Full implementation with wind chain management. Supports in-guard/thunk/out-guard calling, wind chain copy/compare for continuations, and proper unwind/rewind semantics. |

### Key Observations

1. **Module file loading** (`scm_resolve_module`, line 297) searches `%load-path` from the root module and falls back to the current directory. If no file is found, it emits a warning.

2. **Autoload** is implemented as a lazy resolution mechanism in `module_search_variable` (lines 142-187): when a symbol is looked up that matches an autoload spec, the module is loaded, added to uses, and the lookup retries.

3. **Binder recursion guard** (`g_in_binder_call`, line 10) prevents infinite recursion when the binder body triggers symbol lookup in the same module.

4. **Public interface** is created lazily by `scm_ensure_public_interface()` (line 445) on first export. It is a separate `SCM_Module` whose `kind` is `'interface` and whose obarray is populated from `module->exports`.

5. **TeXmacs usage** (test/module/texmacs/): The `texmacs-module` macro at line 27 of boot.scm uses `define-module` with TeXmacs-specific options (`:use`, `:inherit`, `:export`) mapped through a `texmacs-module` macro, plus calls `module-use!` to include the texmacs-user module. The `inherit-modules` macro at line 11 uses `resolve-module` + `module-ref` with `'%module-public-interface` + `module-map` to enumerate exports, then calls `use-modules` + `re-export`.

6. **Module registry** uses `equal?` comparison (since module names are lists of symbols, not atoms), while the obarray uses `eq?` (symbol identity). The registry is lazily initialized.

### Comparison Operations

All support chained comparison for 3+ arguments (e.g., `(< a b c)` means `a < b` and `b < c`). All return `#t` with zero or one argument.

| Operation | Scheme Name | C Function | Status | Notes |
|-----------|-------------|------------|--------|-------|

| `<` | `<` | `scm_c_lt_chain` | present | Uses `BinaryOperator<LtOp<bool, int64_t>>` template. **Potential bug**: float comparison truncates to `int64_t` before comparing due to template specialization, giving incorrect results for negative non-integer floats. |

**Bug note for `<`, `>`, `<=`, `>=`**: The `BinaryOperator` template's float-promotion path calls `Op::run(d_lhs, d_rhs)` with `Op` = `LtOp<bool, int64_t>`, which takes `int64_t` parameters. The `=` operator is not affected because `scm_c_eq_number` handles float promotion directly without the template.

## Macro System

Source: `src/c/macro.cc` (~360 lines), `src/c/eval.cc` (macro expansion interleaved with eval, lines 299-323).

### Overview

pscm macros are traditional Lisp-style explicit-renaming macros defined via `define-macro`. There are two forms:
- `(define-macro name transformer)` — binds a procedure as the macro transformer
- `(define-macro (name args...) body...)` — shorthand that creates a procedure from the signature

Macro expansion happens at evaluation time (not as a separate pre-processing pass). In `eval_with_env()`, before checking special forms, the evaluator checks if the car of a pair is a symbol bound to a macro value. If so, it expands the call, recursively expands any remaining macros in the result, and re-enters the main evaluation loop.

### Macro Expansion Flow Detail

1. `eval_with_env()` receives a list expression
2. At `eval.cc:299-300`, checks if `l->data` is a symbol and `scm_env_exist(env, sym)` returns a macro value
3. If yes, calls `expand_macro_call(env, macro, l->next, ast)` (macro.cc:5-148):
   - Creates a macro environment child of `macro->env` (definition environment)
   - Binds formal parameters to actual arguments as-is (syntax objects, NOT evaluated)
   - Handles rest parameters via `is_dotted` flag
   - Calls `eval_with_list(macro_env, macro->transformer->body)` to evaluate the macro body
   - Copies source location from original call to expansion
4. Calls `expand_macros(env, expanded)` to recursively expand the result (macro.cc:151-228)
5. Re-enters eval loop via `goto entry` to handle the expanded form as a new expression (eval.cc:322)

### Key Observations

1. **Top-down expansion**: The outer macro is always expanded first. If the result contains more macro calls, they are expanded recursively. This is the standard behavior for Lisp-style macros.

2. **Syntax objects are raw AST**: Unevaluated arguments are passed as the original parsed SCM objects (symbols, lists, etc.), not as syntax objects with lexical context. This is the "explicit renaming" approach — macro writers must use `gensym` to avoid variable capture.

3. **Expansion-time vs compile-time**: pscm does macro expansion at evaluation time (not as a separate compile-time phase). There is no separate macro expansion pass before evaluation — expansion is interleaved with evaluation.

4. **No `define-macro` inside `lambda`/`let` at parent scope**: Since `expand_macros()` and `eval.cc` both use `scm_env_exist()` (no parent search), macros defined in surrounding lexical scopes may not be found during expansion. This is a potential correctness issue for lexically-scoped macros.

### Key Observations

1. **Global stdout swap in write/display to port**: The `write` and `display` C functions swap the global `stdout` to a temporary file, print the object, then restore `stdout`. This is fragile — any concurrent output or error during the write goes to the temp file, and the 4096-byte read buffer means large outputs are silently truncated.

2. **String I/O is well-implemented**: String ports use dynamic buffers and proper memory management. `get-output-string` correctly returns the accumulated content and the port remains usable after retrieval.

3. **Soft ports follow Guile 1.8 API**: The vector indexing (write-char=0, write-string=1, flush=2, read-char=3, close=4, read-line=5) matches Guile 1.8. The modes string provides runtime capability checking.

4. **No port encoding/decoding**: Guile 1.8 supports port encoding (latin-1, utf-8, etc.). pscm treats all I/O as raw bytes. This is acceptable for R4RS/R5RS compatibility.

5. **Peek-char lacks soft port support**: `peek_char_from_port` only handles file and string ports. Soft port peek falls through to returning EOF.

### Key Observations

1. **No shared structure support**: The parser cannot handle `#n=` (label) or `#n#` (reference) notation. Cyclic data structures will cause infinite loops. This limits compatibility with code that uses these features for compact representation.

2. **No `\xNN` hex escapes in strings**: Only standard named escapes (`\n`, `\t`, `\r`, `\\`, `\"`) are supported. Hex escape sequences are not parsed — they produce the literal characters `\x`, `N`, `N`.

3. **Keyword implementation is straightforward**: `#:key` is parsed as a symbol named `:key`. The printer checks for the leading `:` and prepends `#`. No keyword type exists — keywords are just symbols with a colon prefix.

4. **Printer uses raw `printf`/`fprintf` throughout**: The printer writes directly to `stdout`/`stderr` via C stdio. There is no port abstraction in `print_ast` — port redirection in `write`/`display` is achieved by swapping the global `stdout` to a temp file. This means `print_ast` cannot be used with arbitrary ports from the C level.

5. **Parser is a single-pass recursive descent**: The parser handles one expression at a time, returning `nullptr` on incomplete input (for `read` from port, the port-level reader pre-buffers a complete expression). Error messages include parse stack traces and evaluation call stacks.

6. **The `read` port implementation has limitations**: `read_expr_from_port()` in `port.cc` uses a heuristic approach (tracking paren depth and string state) to find expression boundaries. It does not handle character literals, vector syntax, or quote shorthand at the port level — it simply finds the top-level expression boundary and delegates to `parse()`. This means the port reader may misidentify boundaries for complex cases.
