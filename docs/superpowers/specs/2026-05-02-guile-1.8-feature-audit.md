## Top Risks for TeXmacs

### 1. GC Stability Under Heavy Load
**Severity**: HIGH
**Description**: The conservative GC has several hard limits: MAX_SEGMENTS=16 (at most 16 heap segments of ~8MB each), MAX_ROOTS=2048 registered root pointers, and GROWTH_SIZE=4MB per segment. When these limits are exhausted, the GC calls abort() (4+ abort sites). The conservative stack scan is inherently imprecise and may retain garbage or miss live references. The stop-the-world mark-sweep design causes full pauses during collection.
**Impact on TeXmacs**: TeXmacs sessions with many open documents or long-running computations could trigger GC exhaustion and crash the editor. Conservative scanning may cause memory leaks over time. GC pauses may cause noticeable UI stutter during collection.

### 2. Missing Phase B/C C API Wrappers
**Severity**: HIGH
**Description**: Six critical module C API functions exist at the internal scm_* level but have no pscm_* wrappers in pscm_api.h: scm_c_resolve_module, scm_c_module_lookup, scm_c_module_define, scm_c_use_module, scm_c_current_module, scm_c_set_current_module. Additionally, there are no pscm_is_*, pscm_list_*, pscm_symbol_*, or pscm_string_* convenience C API functions. The scm_call_0/1/2/3 functions exist internally but have no pscm_* wrappers.
**Impact on TeXmacs**: TeXmacs C++ code that relies on these C API functions will not link or compile against pscm. TeXmacs must either use Scheme-level calls for module operations (adding overhead) or these wrappers must be implemented.

### 3. Numeric Comparison Bugs (Float Truncation)
**Severity**: HIGH
**Description**: The comparison operators (<, >, <=, >=) use a `BinaryOperator` template that truncates double values to `int64_t` before comparison when both operands are FLOAT type. This causes incorrect results for negative non-integer floats. For example, `(< -1.5 -1.0)` incorrectly returns `#f`. The `=` operator is not affected. Additionally, ratio arithmetic (+/-) and ratio comparisons are not properly handled.
**Impact on TeXmacs**: Any TeXmacs code relying on float comparisons could produce silently wrong results. Numeric code involving both positive and negative floats is especially vulnerable. The impact is potentially subtle and hard to debug.

### 4. Missing syntax-rules / Hygienic Macros
**Severity**: MEDIUM
**Description**: pscm only supports traditional `define-macro` (explicit-renaming) macros. There is no `syntax-rules`, `syntax-case`, or `define-syntax`. Existing explicit-renaming macros may suffer from unintended variable capture or hygiene violations.
**Impact on TeXmacs**: Any TeXmacs code or libraries that use hygienic macro systems will not work. Macro-heavy code bases would need to be ported to explicit-renaming style.

### 5. Missing Module C API (pscm_c_* wrappers)
**Severity**: MEDIUM
**Description**: All six Phase B module API functions (pscm_c_resolve_module, pscm_c_module_lookup, pscm_c_module_define, pscm_c_current_module, pscm_c_set_current_module, pscm_c_use_module) are missing as pscm_* public API entry points. The internal scm_* implementations exist but are not accessible from embedding C code.
**Impact on TeXmacs**: TeXmacs code that needs to interact with the module system from C (e.g., creating modules, resolving imports, looking up module bindings) cannot do so without going through the slower Scheme eval path or implementing the wrappers.

### 6. Printer Buffer Limits (4096-byte tmpfile)
**Severity**: MEDIUM
**Description**: The `write` and `display` functions redirect `stdout` to a temp file, print the object, then read back at most 4096 bytes. Outputs larger than 4096 bytes are silently truncated. The global stdout swap is not thread-safe and may cause data races.
**Impact on TeXmacs**: Any TeXmacs code that writes large data structures (e.g., large S-expressions, serialized documents) will have output silently truncated. This could lead to data loss when saving documents or communicating with embedded Scheme.

### 7. Macro Expansion Inconsistency (scm_env_exist vs scm_env_search)
**Severity**: MEDIUM
**Description**: `expand_macros()` and the eval-time macro check both use `scm_env_exist()` which only searches the immediate environment without parent/module/global search. In contrast, `macroexpand-1` uses `scm_env_search()` with full chain search. This means macros defined in parent scopes may be missed during expansion.
**Impact on TeXmacs**: Macros defined in nested `let` or `lambda` scopes may not be found during macro expansion, leading to unexpected "undefined variable" errors.

### 8. for-each MAX_FOR_EACH_ARGS = 10 Limit
**Severity**: LOW
**Description**: The `for-each` special form has a hard-coded limit of 10 argument lists. Attempting to pass more than 10 lists produces a runtime error. This limit does not apply to `map`.
**Impact on TeXmacs**: Unlikely to be triggered in normal usage. Only affects code that iterates over more than 10 lists simultaneously with for-each.

### Key Observations

1. **Module file loading** (`scm_resolve_module`, line 297) searches `%load-path` from the root module and falls back to the current directory. If no file is found, it emits a warning.

2. **Autoload** is implemented as a lazy resolution mechanism in `module_search_variable` (lines 142-187): when a symbol is looked up that matches an autoload spec, the module is loaded, added to uses, and the lookup retries.

3. **Binder recursion guard** (`g_in_binder_call`, line 10) prevents infinite recursion when the binder body triggers symbol lookup in the same module.

4. **Public interface** is created lazily by `scm_ensure_public_interface()` (line 445) on first export. It is a separate `SCM_Module` whose `kind` is `'interface` and whose obarray is populated from `module->exports`.

5. **TeXmacs usage** (test/module/texmacs/): The `texmacs-module` macro at line 27 of boot.scm uses `define-module` with TeXmacs-specific options (`:use`, `:inherit`, `:export`) mapped through a `texmacs-module` macro, plus calls `module-use!` to include the texmacs-user module. The `inherit-modules` macro at line 11 uses `resolve-module` + `module-ref` with `'%module-public-interface` + `module-map` to enumerate exports, then calls `use-modules` + `re-export`.

6. **Module registry** uses `equal?` comparison (since module names are lists of symbols, not atoms), while the obarray uses `eq?` (symbol identity). The registry is lazily initialized.

## Macro System

Source: `src/c/macro.cc` (~360 lines), `src/c/eval.cc` (macro expansion interleaved with eval, lines 299-323).

### Overview

pscm macros are traditional Lisp-style explicit-renaming macros defined via `define-macro`. There are two forms:
- `(define-macro name transformer)` â€” binds a procedure as the macro transformer
- `(define-macro (name args...) body...)` â€” shorthand that creates a procedure from the signature

Macro expansion happens at evaluation time (not as a separate pre-processing pass). In `eval_with_env()`, before checking special forms, the evaluator checks if the car of a pair is a symbol bound to a macro value. If so, it expands the call, recursively expands any remaining macros in the result, and re-enters the main evaluation loop.

### Macro Expansion Flow Detail

1. `eval_with_env()` receives a list expression
2. At `eval.cc:299-300`, checks if `l->data` is a symbol and `scm_env_exist(env, sym)` returns a macro value
3. If yes, calls `expand_macro_call(env, macro, l->next, ast)` (macro.cc:5-148):
   - Creates a macro environment child of `macro->env` (definition environment)
   - Binds formal parameters to actual arguments as-is (syntax objects, NOT evaluated)
   - Handles rest parameters via `is_dotted` flag
   - Calls `eval_with_list(macro_env, macro->transformer->body)` to evaluate the macro body
   - Copies source location from original call to expansion
4. Calls `expand_macros(env, expanded)` to recursively expand the result (macro.cc:151-228)
5. Re-enters eval loop via `goto entry` to handle the expanded form as a new expression (eval.cc:322)

### Key Observations

1. **Top-down expansion**: The outer macro is always expanded first. If the result contains more macro calls, they are expanded recursively. This is the standard behavior for Lisp-style macros.

2. **Syntax objects are raw AST**: Unevaluated arguments are passed as the original parsed SCM objects (symbols, lists, etc.), not as syntax objects with lexical context. This is the "explicit renaming" approach â€” macro writers must use `gensym` to avoid variable capture.

3. **Expansion-time vs compile-time**: pscm does macro expansion at evaluation time (not as a separate compile-time phase). There is no separate macro expansion pass before evaluation â€” expansion is interleaved with evaluation.

4. **No `define-macro` inside `lambda`/`let` at parent scope**: Since `expand_macros()` and `eval.cc` both use `scm_env_exist()` (no parent search), macros defined in surrounding lexical scopes may not be found during expansion. This is a potential correctness issue for lexically-scoped macros.

### Key Observations

1. **Global stdout swap in write/display to port**: The `write` and `display` C functions swap the global `stdout` to a temporary file, print the object, then restore `stdout`. ### Key Observations

1. **No shared structure support**: The parser cannot handle `#n=` (label) or `#n#` (reference) notation. Cyclic data structures will cause infinite loops. This limits compatibility with code that uses these features for compact representation.

2. **No `\xNN` hex escapes in strings**: Only standard named escapes (`\n`, `\t`, `\r`, `\\`, `\"`) are supported. Hex escape sequences are not parsed â€” they produce the literal characters `\x`, `N`, `N`.

3. **Keyword implementation is straightforward**: `#:key` is parsed as a symbol named `:key`. The printer checks for the leading `:` and prepends `#`. No keyword type exists â€” keywords are just symbols with a colon prefix.

4. **Printer uses raw `printf`/`fprintf` throughout**: The printer writes directly to `stdout`/`stderr` via C stdio. There is no port abstraction in `print_ast` â€” port redirection in `write`/`display` is achieved by swapping the global `stdout` to a temp file. This means `print_ast` cannot be used with arbitrary ports from the C level.

5. **Parser is a single-pass recursive descent**: The parser handles one expression at a time, returning `nullptr` on incomplete input (for `read` from port, the port-level reader pre-buffers a complete expression). Error messages include parse stack traces and evaluation call stacks.

6. **The `read` port implementation has limitations**: `read_expr_from_port()` in `port.cc` uses a heuristic approach (tracking paren depth and string state) to find expression boundaries. It does not handle character literals, vector syntax, or quote shorthand at the port level â€” it simply finds the top-level expression boundary and delegates to `parse()`. This means the port reader may misidentify boundaries for complex cases.

### Known Limitations

1. **setjmp/longjmp vs C++ destructors**: The continuation implementation uses C `setjmp`/`longjmp`, which does not unwind C++ stack frames. C++ objects with destructors on the stack between the continuation capture and invocation points will not have their destructors called. This is a fundamental limitation of the approach and is acknowledged in the code (ASan suppression comments). Objects relying on RAII (e.g., `std::string`, `std::vector`, smart pointers) may leak or cause undefined behavior when a continuation is invoked across their scope.

2. **Stack growth heuristic**: The `grow_stack` function (line 123) allocates a fixed 100-element array. This may not be sufficient for very deep stacks where the continuation's saved stack data is much larger than 100 `long` elements above the current frame.

3. **Heap-allocated stack data**: Continuation stack data is allocated with `malloc` (line 36) and freed in `sweep_cont` (gc.cc:741-746). This means each continuation capture does a heap allocation proportional to the current stack depth, which could be expensive for deep stacks.

4. **Wind chain pointer comparison**: `find_common_prefix` (dynwind.cc:29-60) compares wind chain entries by pointer identity (`c1->data == c2->data`). This works because wind chain entries are the original list nodes created by `dynamic-wind`, but the `copy_wind_chain` function creates new nodes with the same `data` pointers, so the comparison is valid only when the original nodes are preserved.

### Correctness

The before/after thunk ordering is correct per R5RS: in_guard is called before the thunk body is entered, out_guard is called after the thunk body exits (both normal and non-local). The wind chain entry is added between the in_guard call and the thunk body, and removed before the out_guard call. This ensures proper ordering: in_guard -> [wind entry added] -> thunk body -> [wind entry removed] -> out_guard.

During non-local exit via continuation (`scm_dynthrow`), the wind chain is properly unwound and rewound. The unwind step calls out_guard for each guard not in the common prefix (working outward from the current wind state), and the rewind step calls in_guard for each new guard (working inward toward the target wind state). This matches Guile 1.8 behavior.

### Lazy Catch Handlers

The `scm_c_with_throw_handler` and `scm_internal_lazy_catch` functions (throw.cc:340-422) register lazy catch handlers as special wind chain entries. These are distinct from normal wind entries and are marked with the symbol `__lazy_catch__`. When `scm_throw` searches the wind chain for handlers, it checks for these lazy catch markers first. If found, the handler is called directly without stack unwinding (no `longjmp`). This is compatible with Guile 1.8's lazy-catch mechanism.

### Known Limitations

1. **Catch/throw does NOT unwind the wind chain**: `scm_throw` (throw.cc:134-191) searches the wind chain for lazy catch handlers but does NOT call out_guard procedures or unwind the wind chain before longjmp. In Guile 1.8, `scm_throw` unwinds the wind chain as part of non-local exit. pscm's catch system relies on the continuation-based wind chain management, which means `throw` without an enclosing `call/cc` may leave the wind chain in an inconsistent state.

2. **Static storage for lazy catch data**: `lazy_catch_storage[100]` (throw.cc:139, 364) is a fixed-size static array. Maximum 100 nested lazy catches.

## GC (Garbage Collector)

Source: `src/c/gc.cc` (~820 lines), `src/c/gc.h` (~134 lines).

### Architecture

The GC is a conservative, stop-the-world mark-sweep collector. It uses a segregated-fit allocation strategy with four fixed-size buckets (16, 32, 64, 128 bytes) plus a large-object bucket for blocks >= 129 bytes. Heap memory is obtained via `mmap` with `MAP_ANONYMOUS`. The initial heap size is 8 MB, growing by 4 MB segments as needed.

### Limits

|-------|-------|--------|-------------|
| MAX_SEGMENTS | 16 | `gc.cc:58` | Maximum 16 heap segments. Total theoretical max heap = 8MB + 15 * 4MB = 68MB (less if any segment is larger due to growth). |
| MAX_ROOTS | 2048 | `gc.h:102` | Maximum 2048 registered root pointers. Calls `abort()` if exceeded. |
| GROWTH_SIZE | 4 MB | `gc.cc:23` | Default segment growth size. Minimum 4 MB per new segment. |
| INITIAL_HEAP_SIZE | 8 MB | `gc.cc:22` | Initial heap allocation. |
| MIN_THRESHOLD | 2 MB | `gc.cc:32` | Minimum GC trigger threshold. |
| GC threshold | 1.5x live | `gc.cc:459` | Next GC triggers when allocated bytes exceed 1.5x the last measured live bytes. |
| Fixed size classes | 16, 32, 64, 128 | `gc.cc:26` | Four size classes for small allocations. |
| GCBlock header | 16 bytes | `gc.h:24` | Overhead per allocation (mark, age, type_tag, size, gc_magic, next_free pointer). |

### Known Limitations

1. **Conservative stack scan is imprecise**: The GC scans the C stack word-by-word looking for values that look like valid heap pointers (range check + alignment + magic check). This can cause false positives (integers that happen to look like pointers), preventing memory from being freed. It can also miss pointers that are in registers or temporary storage.

2. **No incremental/concurrent GC**: The mark-sweep is stop-the-world -- all evaluation is paused during collection. For interactive applications, this may cause noticeable pauses.

3. **Hard heap limits**: With MAX_SEGMENTS=16, the heap can grow to at most ~68 MB. Embedding applications with larger memory requirements will crash with `abort()`.

4. **SMOB tracing is a no-op**: `trace_smob` (gc.cc:749) is empty, meaning any SCM pointers stored inside SMOB data will not be traced and may be collected prematurely. The SMOB mark/sweep/free hooks in `smob.h` are designed for external resource management but the default implementations do nothing useful -- the default `free` function has a notable comment: "Free external memory (when GC is implemented)" (smob.cc:27), indicating the SMOB system predates the current GC and has not been fully integrated.

5. **Port sweep guards against closing stdin/stdout/stderr**: The sweep_port function (gc.cc:719-737) has an explicit guard (`port->file != stdin && port->file != stdout && port->file != stderr`) to prevent closing standard streams. This was a confirmed bug that has since been fixed.

6. **No generational or young-gen optimization**: All objects are treated equally. No generational hypothesis optimization.

### C API Hooks (scm_set_smob_*)

The SMOB system provides C-level hooks for customizing behavior:

- **mark**: Called during GC mark phase to trace SCM pointers within the SMOB's external data. The current GC trace for SMOB (`trace_smob` at gc.cc:749) is a no-op, so custom mark functions are NOT called by the GC.
- **free**: Called when the SMOB object is reclaimed by GC. The default implementation checks `size > 0` and attempts to free external data but the implementation is commented out.
- **print**: Called during `write`/`display` to produce the SMOB's printed representation. Receives the SCM object, port, and `scm_print_state` struct.
- **equalp**: Called by `equal?` to compare two SMOB instances.
- **apply_0/1/2/3**: Called when the SMOB is used as a procedure. Not yet integrated into the evaluator.

### Known Limitations

1. **GC integration incomplete**: The `trace_smob` function is a no-op, so custom mark functions registered via `scm_set_smob_mark` are never called. Any SCM pointers embedded inside SMOB data will be invisible to the GC and may be collected prematurely. The default free function comment ("when GC is implemented") further confirms this.

2. **Scheme-level SMOB API limited**: Only `scm-make-smob-type`, `scm-make-smob`, `scm-smob?`, and `scm-smob-data` are exposed to Scheme. There are no Scheme-level `scm-set-smob-*` functions for setting hooks. The `scm_set_smob_mark`/`free`/`print`/`equalp`/`apply` functions are C API only.

3. **Print function ignores port**: `default_smob_print` (smob.cc:33-52) prints to `stdout` via `printf` and ignores the `port` argument. Custom print functions may also need to use `printf` rather than the port abstraction.

4. **Size>0 data access**: `scm_c_smob_data` (Scheme wrapper) errors when the SMOB has `size > 0`, because it cannot convert a `void*` pointer to a Scheme value. This means Scheme code cannot access the data of SMOBs with external memory.

| `backtrace` | boolean | `#f` | Show backtrace on error. |
| `debug` | boolean | `#f` | Use the debugging evaluator. |
| `stack` | integer | 20000 | Stack size limit (words; 0 = no check). |
| `show-file-name` | SCM | `#t` | Show file names and line numbers in backtraces. `#t` = full, `base` = base names only. |
| `warn-deprecated` | boolean | `#f` | Warn when deprecated features are used. |

### Known Limitations

1. **Options are global state**: Debug options are stored in a static C array. There is no per-module or per-thread debug state. This matches Guile 1.8 behavior (which also uses global state) but limits concurrent usage.

2. **Macros defined via eval_string**: The debug-set!/debug-enable/debug-disable macros are defined at C level by calling `scm_c_eval_string` with Scheme source strings. This means they depend on the evaluator being fully initialized and the macro system being operational. If debug initialization happens before macro system setup, these macros will fail silently (the error is printed but the init continues).
